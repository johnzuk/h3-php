# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A PHP extension written in C that wraps Uber's [H3](https://github.com/uber/h3) geospatial indexing library (v4.x) in an OOP-style API under the `H3\` namespace. The extension links against the system-installed `libh3` shared library.

- PHP: `^8.1` (CI matrix runs 8.1–8.5)
- H3 C library: `^4.0` (CI pins `v4.2.1`)

## Build / test / install

The extension uses the standard PHPize toolchain — there is no composer/npm layer.

```bash
phpize
./configure --with-h3        # auto-discovers libh3 in /usr/local or /usr
make
sudo make install            # installs h3.so into the PHP extension dir
```

Run the test suite (PHPT format, executed by PHP's own `run-tests.php`):

```bash
export NO_INTERACTION=1
export REPORT_EXIT_STATUS=1
export TEST_PHP_EXECUTABLE=$(which php)
php run-tests.php --show-diff -d extension=h3.so ./tests/*.phpt
```

Run a single test:

```bash
php run-tests.php --show-diff -d extension=h3.so ./tests/fn_compact.phpt
```

`run-tests.php` is fetched/generated by `make test` and is gitignored — run `make test` once (or `phpize && ./configure && make` again) if it's missing.

C code is formatted with clang-format (see `.clang-format`: LLVM base, 4-space indent, Linux braces, no column limit).

## Architecture

The extension has a small, flat structure — three files are load-bearing:

- **`h3.stub.php`** — the authoritative declaration of the PHP-facing API (functions, classes, methods, types, `@throws`). Edit this when adding/changing exposed surface.
- **`h3_arginfo.h`** — **generated** from `h3.stub.php` by PHP's `gen_stub.php` tool. Do not hand-edit; regenerate after changing the stub. The generated file's header (`Stub hash: h3v4-upgrade`) is the marker used to detect drift.
- **`h3.c`** — the C implementation. Every PHP function/method declared in the stub has a `PHP_FUNCTION` / `PHP_METHOD` here. Adding a new function means: declare it in the stub, regenerate `h3_arginfo.h`, then implement it in `h3.c`.

### Bridging conventions in `h3.c`

The first ~200 lines establish the conversion helpers that the rest of the file relies on heavily:

- `obj_to_h3` / `h3_to_obj` — read/write the `H3Index::$index` long property to go between a 64-bit `H3Index` and a PHP object. Same pattern for `H3DirectedEdge`.
- `obj_to_geo` / `geo_to_obj` — `LatLng` objects store degrees in PHP but the H3 C API expects radians; conversion happens at the boundary.
- `h3_array_to_zend_array` / `zend_array_to_h3_array` — bulk conversion between C `H3Index[]` and PHP arrays of `H3Index` objects. Invalid (zero) indexes are filtered out on the way back to PHP.
- Class entries are global (`H3_H3Index_ce`, etc.) and registered in `PHP_MINIT_FUNCTION(h3)`.

### Validation toggles (INI)

Two `php.ini` settings gate runtime checks:

- `h3.validate_res` (default `On`) — checked by the `VALIDATE_H3_RES` macro before any resolution-taking call. Throws `H3\H3Exception` with code `H3_ERR_CODE_INVALID_RES`.
- `h3.validate_index` (default `Off`) — checked by `VALIDATE_H3_INDEX` / `VALIDATE_H3_UNI_EDGE`. Off by default for performance; turn on for debugging.

Tests that exercise the "invalid input → exception" path typically set these via `--INI--` blocks in the `.phpt` file.

### Error model

All extension-thrown exceptions are `H3\H3Exception` (extends `\RuntimeException`), with integer codes defined both in `h3.c` (`H3_ERR_CODE_*`) and re-declared in userland `h3_consts.php`. `h3_consts.php` also exposes the H3 v4 native error codes (`H3_E_*`) as plain PHP constants; it is informational userland code, not loaded by the extension itself.

### Tests

`tests/*.phpt` are PHPT files following PHP's standard format (`--TEST--`, `--EXTENSIONS--`, `--FILE--`, `--EXPECT--`/`--EXPECTF--`). Naming convention:

- `Class_method.phpt` — method tests (e.g. `H3Index_kRing.phpt`)
- `fn_function.phpt` — namespaced function tests (e.g. `fn_compact.phpt`)
- `*_args.phpt` — argument validation / error path tests
- `*_errors.phpt` — exception path tests

Every test starts with `--EXTENSIONS--\nh3` so it only runs when the extension is loaded.

## CI

Two GitHub Actions workflows in `.github/workflows/`:

- `build-and-test.yml` — runs on every push; builds H3 from source then the extension, across PHP 8.1–8.5.
- `release.yml` — triggered by GitHub releases; same matrix on both Linux and macOS, builds artifacts for distribution.

Both pin H3 to `v4.2.1` via matrix variable — bump it there when upgrading the underlying H3 library.
